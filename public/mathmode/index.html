Portfolio: Math Input

The Project

Wolfram|Alpha is a product that does really quite an astonishing number of things…but what users truly do with it is math. Which was an issue for growing the product—you could enter quite complex mathematical inputs, but they either had to be in “natural language”, which is frankly not that natural when trying to do an integral…

integrate sin x dx from x=0 to pi

…or you could type it in Mathematica/Wolfram Language syntax, which is quite programmer-y. A 4×4 matrix would become…

[{2, 1, 0, -3}, {3, -1, 0, 1}, {1, 4, -2, -5}]

Which is simply not user-friendly, especially for the K–12 market.

So, Math Input was developed to help users better do their job to be done.

The Goals

  • User friendly. Create an easy-to-use interface for math students to work with templates for common equations in algebra, calculus, matrices, and trigonometry.
  • Make it pop. Give it a little visual zazz to help differentiate it from the existing functionality, and make it look less “technical” and intimidating compared to some math education products.
  • School laptop friendly. Make it mobile-friendly, as much as possible, but make it extremely usable on smaller laptop viewports for Chromebooks for education.
  • Systemitize. Create a design system that keeps colors coherent and systemitized, for the existing color library has gotten rather Byzantine.

The Constraints

  • Domain-specific knowledge. Though I have some math education, it’s been a while since freshman year calculus, so I had to steel myself to re-learn some of these equations and symbols. (A Dirac delta function? Huh?)
  • Limited UX guidance. Due to a variety of team bandwidth limitations, I was responsible for signifcant UX/UI decicions, especially for mobile viewports.
  • Mobile issues. It was important to make this work decently well on phones, and very well on relatively small viewport Chromebooks, to target the education market. Which is difficult, given the potential density of these equation templates.

The Process

  • Prototyping. There were so many micro-interactions and edge cases where creating a functional, high-fideilty prototype really helped clarify a lot of aspects of this design and user flow. Such as…

      The little phone detail

    • A little phone detail. I’m just a bit proud of this particular detail. Navigating within templates on phones presented a lot of issues—complex templates like sums and products resulted in small, dense touch targets. I eventually landed on what I called a “dpad” in my head. A nice little affordance that makes it easy to step sequentially through a template without having to zoom & tap.
    • Math typesetting. Showing developers how mathematical symbols, like integrals, should scale vertically by using CSS and SVGs was quite useful, and also guided the way to me providing ready-to-go SVGs for final implementation.
    • Grid layout. Using CSS grid to prototype the layout of all these various template icons helped moved things along—went through quite a few iterations of the template “keyboard” layout, and being able to quickly change some grid settings versus manual layout was much faster.
    • Animations. Crafting some CSS transitions and animations in the browser helped clarify certain transitions for both stakeholders and developers.
  • Utilizing corporate knowledge. Wolfram’s primary product is called Mathematica. So, as one might infer: quite a few math folks on staff. Whenever a question about math would come up—the vagaries of equation typsetting, order of operations, suggestions as to icons for particular functions—I reached out for assistance from the team.
